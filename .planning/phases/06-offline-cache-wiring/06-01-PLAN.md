(RoutingService.calculateRoute(fromPlace:toPlace:mode:tripID:context:)) so that SwiftData-cached
routes are served to users when offline.

MapViewModel. The coordinate-only overload is called instead, bypassing L1+L2 cache entirely.

- TripMapView injects modelContext into the VM on appear
- calculateDirectionRoute(to:) branches: place-UUID overload when Place is selected, coordinate
  overload as fallback (search items, AI results, place-to-place with missing context)

IMPORTANT — the place-UUID overload signature is `fromPlace origin: Place, toPlace destination: Place`. The variable `place` in `calculateDirectionRoute(to place: Place)` IS the destination. The origin is NOT a Place when `routeFromCurrentLocation == true` (it's GPS).

Correct branch logic:

```swift
let results: [RouteResult]
if let ctx = modelContext, !routeFromCurrentLocation, let originPlace = routeOriginPlace {
    // Place-to-Place: use UUID overload with cache
    results = await RoutingService.shared.calculateRoute(
        fromPlace: originPlace,
        toPlace: place,
        mode: selectedTransportMode,
        tripID: trip.id,
        context: ctx
    )
} else if let ctx = modelContext, routeFromCurrentLocation {
    // GPS origin → Place destination: also use UUID overload (origin is GPS but dest has UUID)
    // RoutingService online path will fetch+cache; offline path: lookup by dest UUID only not possible
    // Use a synthetic "GPS origin" approach: call coordinate overload but do L2 lookup by dest UUID
    // SIMPLEST CORRECT FIX: For GPS→Place, call coordinate overload online,
    // but for offline prefer L2 lookup using dest UUID via RoutingCacheService directly.
    if await OfflineCacheManager.shared.isOnline {
        results = await RoutingService.shared.calculateRoute(
            from: origin, to: destination, mode: selectedTransportMode
        )
    } else {
        // Offline: try L2 cache by destination UUID (any origin → place)
```

STOP. The above is over-engineered. The audit evidence says:
"Users who tap 'Подготовить офлайн' → routes cached → go offline → tap place → 'Маршрут' → get error"

`preCacheDay` stores routes keyed by `place.id` pairs (origin UUID + destination UUID). The user selects a destination Place from their trip. The origin in `calculateDirectionRoute` may be GPS or another Place.

The place-UUID overload requires BOTH origin and destination as Place objects. But GPS origin is a `CLLocationCoordinate2D`, not a Place with a UUID.

Re-read `RoutingCacheService.lookup` — it requires `origin: UUID`. GPS has no UUID.

THE CORRECT MINIMAL FIX based on `preCacheDay`:
- `OfflineCacheManager.preCacheDay` caches routes between consecutive Places in a day (Place[i] → Place[i+1])
- `calculateDirectionRoute` is called when user selects a Place (destination) from the trip
- When `routeOriginPlace != nil` (user selected a specific origin Place): call place-UUID overload with both UUIDs
- When `routeFromCurrentLocation == true`: GPS origin has no UUID → cannot use L2 cache by UUID pair → fall through to coordinate overload (will return `[]` offline → shows "Маршрут не найден")

Actually re-reading `RoutingService.calculateRoute(fromPlace:toPlace:)`:
- Online: fetches + caches by (origin.id, dest.id)
- Offline: `RoutingCacheService.lookup(origin: origin.id, dest: dest.id, mode:, context:)`

If user's origin is GPS location (no UUID), we cannot use the place-UUID overload. The cache was stored with Place UUIDs. This means GPS→Place offline is still broken.

BUT: the typical user flow that needs fixing is: user browses trip places → taps "Маршрут" between two trip places (routeOriginPlaceID is set) OR sets "from current location" and destination is a trip place.

For `routeFromCurrentLocation = true`, the simplest approach is: call the coordinate-only overload online, but ALSO check L2 cache by destination UUID (ignore origin UUID) if offline. However, RoutingCacheService.lookup requires both UUIDs.

SIMPLEST CORRECT IMPLEMENTATION (matches audit fix description exactly):
> "Wire MapViewModel.calculateDirectionRoute(to:) to call the place-UUID overload when selectedPlace and tripID are available, passing modelContext for SwiftData access."

Meaning: when `modelContext` is available AND the destination is a trip Place:

```swift
let results: [RouteResult]
if let ctx = modelContext, let originPlace = routeOriginPlace {
    // Both origin and destination are trip Places — use UUID overload with cache
    results = await RoutingService.shared.calculateRoute(
        fromPlace: originPlace,
        toPlace: place,
        mode: selectedTransportMode,
        tripID: trip.id,
        context: ctx
    )
} else {
    // GPS origin or no modelContext — coordinate overload (no offline cache)
    results = await RoutingService.shared.calculateRoute(
        from: origin, to: destination, mode: selectedTransportMode
    )
}
```

This is the minimal correct fix. GPS→Place routing offline remains unavailable (pre-cache stores Place→Place pairs, GPS has no UUID). The audit success criterion is: "MapViewModel.calculateDirectionRoute(to:) calls the place-UUID overload with tripID and modelContext when a Place is selected" — which this satisfies when `routeOriginPlace` is set.
