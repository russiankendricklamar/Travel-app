## Executive Summary

Phase 5 gap closure fixed the `offlineNoCacheMessage` reachability and `navigationSteps` in auto-cached routes. However, the integration checker found a **new critical gap**: the place-UUID routing overload with L2 cache support (`calculateRoute(fromPlace:...)`) is **never called** from `MapViewModel`. All user-initiated routing goes through the coordinate-only overload which has no SwiftData cache lookup. Pre-cached routes exist in the database but are unreachable from the primary user flow.

### OFFL-02 Gap Detail

**Root cause:** `MapViewModel.calculateDirectionRoute(to:)` calls `RoutingService.shared.calculateRoute(from:to:mode:)` — the coordinate-only overload at RoutingService.swift line ~138. This overload has NO L2 SwiftData cache lookup.

The place-UUID overload `calculateRoute(fromPlace:toPlace:mode:tripID:context:)` at lines 196-248 contains the full L1+L2 offline cache logic but has **zero call sites** outside RoutingService.swift itself.

**Impact:** Users who tap "Подготовить офлайн" → routes cached in SwiftData → go offline → tap place → "Маршрут" → get "Маршрут недоступен офлайн" despite routes being in the database.

**Fix:** Wire `MapViewModel.calculateDirectionRoute(to:)` to call the place-UUID overload when `selectedPlace` and `tripID` are available, passing `modelContext` for SwiftData access.
